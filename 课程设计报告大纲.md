## 1. 选题背景与目标（约 500 字）

### 1.1 问题背景
- 信息过载是现代社会痛点——用户每天面对海量资讯，难以高效获取关键信息
- 传统 RSS 阅读器只做聚合，缺乏智能摘要和分类能力
- 自动文本摘要技术的两类路线：抽取式 vs 生成式

### 1.2 项目目标
- 构建端到端资讯摘要订阅系统：自动抓取 → 摘要 → 分类 → Web 展示
- 实现并对比 TextRank 抽取式与 LLM 生成式两种摘要方案
- 支持多信息源（Hacker News + 36氪）+ 多长度档位（一句话/100字/200字）
- 交付可运行的 Web 应用，展示至少 3 天摘要数据

### 1.3 涉及课程知识
| 模块 | 在项目中的应用 |
|------|---------------|
| 词向量 / TF-IDF | TextRank 句子向量化 |
| Seq2Seq / Attention | LLM 生成式摘要的底层原理 |
| Transformer | Qwen2.5 模型架构基础 |
| Prompt Engineering | 摘要生成的 System Prompt 设计 |
| 结构化输出 | LLM 输出 JSON（分类+摘要） |

## 2. 相关技术介绍（约 800 字）

### 2.1 TextRank 抽取式摘要
- 源自 PageRank 算法：将句子视为图的节点，句子相似度作为边权重
- 核心步骤：分词 → TF-IDF 向量化 → 余弦相似度矩阵 → PageRank 迭代 → 选 Top-N 句
- 优点：无监督、不依赖训练数据、速度快
- 缺点：无法生成新文本，摘要是原文片段的拼接

### 2.2 LLM 生成式摘要
- 基于 Transformer Decoder 架构的自回归文本生成
- Qwen2.5 模型：阿里通义千问系列，7B 参数，支持中英双语
- 通过 Prompt Engineering 控制输出：分类 + 三种长度摘要 → 结构化 JSON
- 优点：理解语义、可重新组织语言
- 缺点：推理速度慢、依赖模型质量

### 2.3 ROUGE 评估指标
- ROUGE-L：基于最长公共子序列（LCS）计算召回率/精确率/F1
- 用于衡量生成摘要与参考摘要的 n-gram 重叠度
- 中文需先分词再计算（本项目使用 jieba）

### 2.4 正文提取
- trafilatura：现代 Web 内容提取库，自动识别正文、过滤广告导航
- 对比 newspaper3k：后者在 Python 3.13 下有依赖兼容问题

## 4. 关键实现细节（约 800 字）

### 4.1 TextRank 自实现

与使用 sumy 等现成库不同，本项目完整实现了 TextRank 算法以体现对原理的理解：

```python
# 核心流程（textrank.py）
1. split_sentences()  # 正则分句
2. tokenize()         # jieba 分词 + 停用词过滤
3. build_tfidf_matrix()  # TF-IDF 句子向量
4. cosine_similarity_matrix()  # 余弦相似度
5. pagerank(d=0.85)   # 迭代至收敛
6. 选 Top-N 句（保持原文顺序）
```

TF-IDF 向量化而非简单的词频统计，使得高频但低信息量的词被降权。PageRank 的阻尼系数 d=0.85 是经典选择。

### 4.2 LLM Prompt 设计

一次 API 调用同时完成分类和三种长度摘要，减少 token 消耗：

```
System: 你是新闻摘要助手。分类到{AI/科技,创业/商业,编程/技术,其他}。
        按 JSON 格式输出：{category, one_sentence, short, long}

User:   文章标题：{title}
        文章正文：{truncated 3000 chars}
```

结构化 JSON 输出使后续处理无需额外解析。规范化步骤将 LLM 输出的非标准分类归为"其他"。

### 4.3 数据抓取策略

- **HN API**：获取 Top Stories ID → 逐个查详情 → trafilatura 提取正文
- **36kr RSS**：lxml 直接解析 XML（避免 feedparser 在 Python 3.13 的兼容问题）→ trafilatura 提取正文
- 降级策略：正文提取失败时，若 RSS 简介 ≥ 100 字则用作正文
- 正文 ≥ 50 字（中文）/ ≥ 100 字（英文）的阈值过滤低质量抓取

### 4.4 数据库设计考量

- SQLite 单文件，零配置，适合课程项目
- WAL 模式提升并发写入性能
- UNIQUE(source, url) 防止重复抓取
- LEFT JOIN 查询即时获取文章+摘要，避免 N+1 问题

## 5. 实验与评估（约 1000 字）

### 5.1 ROUGE-L 评估（20 篇文章）

| 方案 | 文章数 | ROUGE-L F1 |
|------|--------|------------|
| TextRank 抽取式 | 20 | **0.1322** |
| LLM 生成式 (100字) | 20 | **0.3571** |
| LLM 生成式 (200字) | 20 | **0.2562** |

**分析**：
- LLM 生成式 ROUGE-L 是 TextRank 的 2.7 倍，差距显著
- LLM 200 字摘要 ROUGE-L（0.2562）低于 100 字（0.3571），因为更长文本与参考摘要的精确匹配概率更低——这是 ROUGE 指标本身偏向短文本的局限性
- TextRank 得分低的原因：抽取式摘要使用原文原句，而人工参考摘要重新组织语言，导致 n-gram 重叠少

### 5.2 人工评分（5 篇文章，1-5 分）

| 维度 | TextRank | LLM |
|------|:---:|:---:|
| 流畅性 | 5.0 | 4.6 |
| 信息覆盖度 | 4.0 | **4.8** |
| 简洁性 | 4.4 | 4.8 |
| **综合** | **4.47** | **4.73** |

**分析**：
- TextRank 流畅性满分——因为它直接摘抄原文，句子本身是通顺的
- LLM 信息覆盖度显著领先（4.8 vs 4.0）——TextRank 容易选到非核心句
- LLM 综合略优（+6%），和 ROUGE 的 170% 差距形成对比，说明 ROUGE 可能低估了抽取式摘要的质量

### 5.3 分类准确率（20 篇文章）

| 类别 | 样本 | 正确 | 准确率 |
|------|:---:|:---:|:---:|
| AI/科技 | 5 | 5 | 100% |
| 创业/商业 | 5 | 5 | 100% |
| 编程/技术 | 5 | 5 | 100% |
| 其他 | 5 | 5 | 100% |
| **总计** | **20** | **20** | **100%** |

LLM 基于 Prompt 的分类在这批数据上完全正确，说明预设的四分类标签与文章实际主题契合。

### 5.4 方案对比总结

| 维度 | TextRank | LLM 生成式 |
|------|----------|------------|
| 实现复杂度 | 80 行自实现 | Prompt 工程 |
| 推理速度 | < 1 秒/篇 | 3-8 秒/篇 |
| ROUGE-L | 0.1322 | **0.3571** |
| 人工评分 | 4.47 | 4.73 |
| 依赖 | jieba + numpy | Ollama + Qwen2.5 |
| 能否离线 | ✅ | ❌（需模型） |

**结论**：LLM 生成式在 ROUGE 和人工评分上均优于 TextRank，但需要 GPU/模型支持。实际部署中可考虑混合策略——快速场景用 TextRank，高质量场景用 LLM。

## 6. 总结与心得（约 500 字）

### 6.1 项目收获
- 完整体验了 NLP 应用从数据获取到评估的全流程
- 在实践中理解了抽取式 vs 生成式摘要的本质差异：不是简单的"好坏"之分，而是"忠实度 vs 可读性"的权衡
- Prompt Engineering 是工程实践中的核心技能——好的 Prompt 一次调用完成多项任务

### 6.2 遇到的问题
- Python 3.13 兼容性：newspaper3k、feedparser 等库在新版本 Python 下有依赖问题，迁移到 trafilatura + lxml 解决
- Ollama SDK bug：ollama Python 库在 Windows 上返回 502，改用 requests 直接调 HTTP API 解决
- 数据库路径：相对路径依赖 CWD，改为基于 config.py 位置的绝对路径

### 6.3 改进方向
- 增加更多信息源（微信公众号、知乎等）
- 引入用户个性化——基于阅读历史调整摘要偏好
- 邮件/微信推送，做成真正的"订阅"体验
- 使用更大模型（如 Qwen2.5:14b）或 API 模型提升摘要质量
