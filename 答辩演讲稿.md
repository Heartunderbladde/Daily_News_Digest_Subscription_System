## 开场（30 秒）— PPT 第 1 页

各位老师、同学大家好。我们做的项目是**每日资讯摘要订阅系统**，属于课程设计题目 4。

一句话概括：它自动从多个信息源抓取新闻，用两种方案生成摘要——传统的 TextRank 抽取式和我们用大语言模型做的生成式——然后自动分类，最后在网页上展示给你看。

## 一、项目概述（1 分钟）— PPT 第 2 页

信息过载是我们每天都会遇到的问题。打开手机，几十条推送、上百篇文章，真正值得看的可能就三五篇。

传统 RSS 阅读器只做聚合，不加工。我们的系统做了三件事：
- **自动抓取**：从 Hacker News 和 36 氪每天获取几十篇文章
- **智能摘要**：用 TextRank 抽取式和 LLM 生成式两套方案，把长文压成一句话、100 字、200 字三档
- **自动分类**：归到 AI/科技、创业/商业、编程/技术、其他四个类别

用户打开网页就能看到今日要闻，不用自己翻。

**涉及课程知识：** TF-IDF、PageRank、Transformer 架构、Prompt Engineering、结构化 JSON 输出、工程实践。覆盖了课程至少 5 个模块的内容。

## 三、技术讲解（5-7 分钟）— PPT 第 5-7 页

### 3.1 系统架构

（展示架构图）

系统分四层。最下面是 SQLite 数据层，两张表——articles 存原始文章，summaries 存摘要，通过 article_id 关联。

往上是业务层。Fetcher 模块负责从 Hacker News API 和 36 氪 RSS 抓取文章，用 trafilatura 提取正文。Summarizer 模块包含两个核心：TextRank 抽取式和我们自实现的、LLM 生成式调用 Qwen2.5。

再往上是 Pipeline 管道层——把抓取和摘要串成一条自动流水线。用户点一下"刷新"，fetch_all 抓文章入库，然后 summarize_all_pending 对没有摘要的文章生成摘要。

最上面是 Streamlit 四页 Web 界面。

总共 14 个 Python 文件，6 个模块，模块间接口清晰。

### 3.2 TextRank 自实现——为什么自己写？

（展示代码片段）

TextRank 是一个经典的抽取式摘要算法，源自 PageRank。

我们的实现分五步：
1. 正则分句
2. jieba 分词加停用词过滤
3. 句子级 TF-IDF 向量化——这是关键。不用简单词频，而是用 TF-IDF，让"的""了"这种高频词被 IDF 降权，真正有语义的实词主导相似度计算
4. 计算句子两两之间的余弦相似度，得到邻接矩阵
5. PageRank 迭代，阻尼系数 0.85，收敛到 10^-6

最后选 Top-K 个句子，按原文顺序排列，不是按得分排——这样摘要读起来还是连贯的。

**为什么不调 sumy 库？** 答辩能讲清楚原理。调库一行代码，但老师一问"TextRank 核心是什么"就答不出。自己实现 80 行，每一步都理解。

### 3.3 LLM Prompt 设计——一次调用完成四项任务

（展示 Prompt 策略）

LLM 用的是 Qwen2.5:7B，通过 Ollama 本地部署，4070 笔记本 8GB 显存刚好跑。

Prompt 设计的核心思路：**一次 API 调用，同时完成分类和三种长度摘要**。

System Prompt 告诉模型：你是新闻摘要助手，分类到这四个标签，按 JSON 格式输出——category、one_sentence、short、long。

Temperature 设 0.3——保证输出稳定，不会每次摘要不一样。

正文截断到前 3000 字符——既足够模型理解文章内容，又不超出上下文窗口。

解析时做两层保护：正则提取 JSON 代码块，然后规范化——如果 LLM 输出了一个不在我们四分类里的标签，统一归到"其他"。

一个坑是 Ollama 的 Python SDK 在 Windows 上返回 502。排查后发现 `curl` 直接调 API 正常，是 SDK 的 HTTP 客户端有问题。我们直接用 `requests` 库替代 SDK，问题解决。

## 四、实验结果（3 分钟）— PPT 第 8-10 页

### 4.1 ROUGE-L 评估

我们为 20 篇文章撰写了人工参考摘要作为 ground truth，计算 ROUGE-L。

（展示数据）

| | TextRank | LLM 100字 | LLM 200字 |
|------|:---:|:---:|:---:|
| ROUGE-L F1 | 0.13 | **0.36** | 0.26 |

LLM 是 TextRank 的 2.7 倍。

但注意一个有意思的现象：LLM 200 字摘要的 ROUGE-L 反而不如 100 字。这不是 200 字更差——它包含更多细节，信息量更大——而是 ROUGE 指标偏向短文本。更长文本与参考摘要的精确 n-gram 匹配概率天然更低。

**这个发现本身就是一个有价值的实验结论：ROUGE 不能作为摘要质量的唯一指标。**

### 4.2 人工评分

所以我们补充了人工评分。5 篇文章，流畅性、信息覆盖度、简洁性三个维度，1 到 5 分。

（展示数据）

| 维度 | TextRank | LLM |
|------|:---:|:---:|
| 流畅性 | **5.0** | 4.6 |
| 信息覆盖度 | 4.0 | **4.8** |
| 简洁性 | 4.4 | 4.8 |
| 综合 | 4.47 | **4.73** |

TextRank 流畅性满分——因为它直接摘抄原文句子。这恰恰暴露了抽取式的本质：它不"写"，它只"搬运"。

LLM 覆盖度领先——4.8 vs 4.0——因为它能归纳概括，而不是机械选句。

最关键的数据对比：人工评分 LLM 只领先 6%，但 ROUGE 领先 170%。这说明 ROUGE 作为表面匹配指标，可能低估了抽取式摘要的实际可读性。

### 4.3 分类准确率 & 方案总结

分类准确率 20 篇测试，**100% 全对**。说明 Qwen2.5:7B 对科技类文章的分类能力完全够用。

（展示对比总表）

总结一下两种方案：LLM 质量更高但需要 GPU、速度 3-8 秒一篇；TextRank 更快更轻但只能搬运原文。实际部署可以混合——实时场景用 TextRank，高质量场景用 LLM。

## 五、遇到的问题与总结（1 分钟）— PPT 第 11-12 页

分享三个核心坑。这些都写在报告里了：

**Python 3.13 兼容性。** newspaper3k 和 feedparser 在新版 Python 下因 sgmllib 移除而无法导入。解决方案是迁移到 trafilatura 做正文提取、lxml 直接解析 RSS XML。

**Ollama SDK 故障。** Python SDK 返回 502，排查一晚发现 curl 正常。改用 requests 直接调 HTTP API 解决。

**数据库路径漂移。** Streamlit 的 CWD 和我们模块位置不一致，导致数据库被创建到两个不同位置，页面一直显示无数据。改成基于 config.py 位置的绝对路径后解决。
