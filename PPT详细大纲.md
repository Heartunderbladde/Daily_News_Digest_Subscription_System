## 第 5 页：关键技术 ① — TextRank 自实现（1.5 分钟）

**标题：** TextRank 抽取式摘要（为什么自己写？）

流程图：
```
原文 → 分句 → jieba分词 → TF-IDF向量 → 余弦相似度矩阵 → PageRank迭代 → 选Top-3句
```

代码片段（放核心 6 行）：
```python
tfidf = build_tfidf_matrix(sentences)   # 句子级 TF-IDF
sim = cosine_similarity_matrix(tfidf)    # 余弦相似度
scores = pagerank(sim, d=0.85)           # PageRank 迭代
top3 = sorted(enumerate(scores), key=lambda x: x[1])[-3:]
return "".join(sentences[i] for i in sorted(top3))
```

**设计决策：为什么不用 sumy？** → 自实现能讲清 TF-IDF + PageRank 原理，答辩有深度。

> 🎤 "TextRank 五步：分句、分词、向量化、算相似度、PageRank 排序选句。核心创新点是用 TF-IDF 做句子向量而非简单词频——这样'的''了'这种高频词被 IDF 降权，真正有语义的实词主导相似度计算。没有调 sumy，自己写了 80 行，答辩能讲清楚每一步。"

## 第 6 页：关键技术 ② — LLM Prompt 设计（1.5 分钟）

**标题：** LLM 生成式摘要（一次调用完成所有任务）

Prompt 策略（放 System Prompt 关键片段）：

```
你是新闻摘要助手。分类到：AI/科技、创业/商业、编程/技术、其他。
必须按 JSON 输出：
{
  "category": "...",
  "one_sentence": "...",   ← 一句话 ≤50 字
  "short": "...",           ← ≈100 字
  "long": "..."             ← ≈200 字
}
```

**设计决策：**
- 一次调用 = 分类 + 3 种长度摘要 → 省 token
- temperature=0.3 → 保证输出稳定
- 正文截断 3000 字符 → 平衡上下文和速度
- 用 requests 直接调 HTTP API（而非 ollama SDK，后者有 bug）

> 🎤 "Prompt 设计是整个 LLM 摘要的核心。一次调用同时完成分类和三种长度摘要——不是调四次，而是一次。System Prompt 约束 JSON 格式，temperature 设低保证稳定。一个坑是 ollama 官方 Python SDK 在 Windows 上返回 502——排查后发现直接调 HTTP API 就正常，所以我们用 requests 替代了 SDK。"

## 第 7 页：关键技术 ③ — 抓取 + 正文提取（1 分钟）

**标题：** 数据抓取策略

| 信息源 | 协议 | 正文提取 | 降级策略 |
|--------|------|----------|----------|
| HN | HTTP API（免费无认证） | trafilatura | 跳过 |
| 36kr | RSS → lxml.etree | trafilatura | 用 RSS 简介 |

**技术选型：**
- trafilatura 而非 newspaper3k → Python 3.13 兼容
- lxml 直接解析 RSS 而非 feedparser → 同上原因
- UNIQUE(source, url) 防重 + 长度阈值过滤低质量

> 🎤 "抓取有两个坑。一是 Python 3.13 移除了 sgmllib，导致 newspaper3k 和 feedparser 都不兼容——我们改用 trafilatura 和 lxml 直接解析。二是正文提取不一定成功——我们有降级策略，RSS 简介够长就用简介。UNIQUE 约束保证同一篇文章不会重复入库。"

## 第 8 页：实验结果 ① — ROUGE-L（1 分钟）

**标题：** ROUGE-L 评估（20 篇文章 vs 人工参考摘要）

大号居中表格：

| | TextRank | LLM (100字) | LLM (200字) |
|------|:---:|:---:|:---:|
| ROUGE-L F1 | **0.13** | **0.36** | 0.26 |

LLM 是 TextRank 的 **2.7 倍**。

分析三点（每点一行）：
- ✅ LLM 重写语言 → 与人工摘要匹配度高
- ⚠️ TextRank 抽原句 → n-gram 重叠天然低
- ⚠️ 200 字比 100 字 ROUGE 更低 → ROUGE 偏向短文本

> 🎤 "ROUGE 评估用了我们自己写的 20 篇人工参考摘要。LLM 是 TextRank 的 2.7 倍。但注意 200 字摘要的 ROUGE 反而不如 100 字——这暴露了 ROUGE 的局限：它倾向于给短文本打高分，所以我们不止看 ROUGE，还做了人工评分。"

## 第 9 页：实验结果 ② — 人工评分 + 分类准确率（1 分钟）

**标题：** 人工评分 & 分类准确率

**人工评分（5 篇，1-5 分）：**

| 维度 | TextRank | LLM |
|------|:---:|:---:|
| 流畅性 | 5.0 | 4.6 |
| 信息覆盖度 | 4.0 | **4.8** |
| 简洁性 | 4.4 | 4.8 |
| **综合** | 4.47 | **4.73** |

**分类准确率：20/20 = 100%**

关键发现（配一句话）：
- TextRank 流畅性满分因为它抽的是人写的原句
- LLM 覆盖度领先 — 能归纳，不是摘抄
- 人工评分差距（+6%）远小于 ROUGE 差距（+170%） → ROUGE 低估了 TextRank

> 🎤 "人工评分揭示了 ROUGE 看不到的东西。TextRank 流畅性满分——因为它抄的就是人写的句子。但信息覆盖度只有 4.0——它有时候会抽到对话片段这种非核心内容。LLM 覆盖度 4.8，差距在归纳能力。分类 100% 全对，说明 Qwen2.5 对科技类文章分类很可靠。最后一点很关键——人工评分 LLM 只领先 6%，但 ROUGE 领先 170%，说明纯靠 ROUGE 评估会低估抽取式摘要。"

## 第 10 页：方案对比总结（30 秒）

**标题：** 两种方案，各有所长

| 维度 | TextRank | LLM 生成式 |
|------|----------|------------|
| ROUGE-L | 0.13 | **0.36** |

> 🎤 "一句话总结：LLM 质量更高但更贵更慢，TextRank 更快更轻但只能搬运原文。实际落地可以混合——实时场景用 TextRank，高质量场景用 LLM。"

## 常见提问准备

| 问题 | 回答思路 |
|------|---------|
| 为什么选 TextRank 而不是其他算法？ | 无监督、Python 可实现、与 LLM 形成互补对比 |
| LLM 摘要会不会产生幻觉？ | temperature=0.3 降低随机性；系统只做摘要不做创造；发现幻觉可降级到 TextRank |
| 数据量增大 10 倍架构怎么调整？ | HN API 并发抓取（asyncio）、TextRank 批处理、LLM 加队列、SQLite 换 PostgreSQL |
| 最难的部分是什么？ | Prompt 工程——调了很多版才让 JSON 输出稳定；Python 3.13 兼容性排查 |
| 课程哪些知识点用到了？ | TF-IDF、PageRank、Transformer、Prompt Engineering、结构化输出、工程实践 |
| 为什么用 Qwen2.5 而不是 GPT？ | 课程要求 Ollama 本地部署；Qwen2.5:7B 在 8GB 显存可跑；免费无 API 费用 |
