# Vibe Coding — 开发过程反思

> 每日资讯摘要订阅系统 | 开发工具：Claude Code | 2026 年 6 月

---

## 一、Vibe Coding 的优势

### 1. 架构决策效率极高

传统开发中，技术选型往往需要查阅大量文档、对比多个方案才能做出决定。Vibe Coding 模式下，AI 在几秒内完成了信息源选择（HN vs 其他 API）、Web 框架（Streamlit vs Gradio）、持久化方案（SQLite vs JSON vs ChromaDB）的对比分析。每个决策 AI 都给出了推荐方案 + 理由 + 替代方案，我只需逐个确认。

**关键优势：** AI 不会遗漏评估维度。例如在选择正文提取库时，AI 主动考虑了 Python 3.13 兼容性——这是人类开发者容易忽略的细节。

### 2. 全栈能力

项目覆盖数据抓取、NLP 算法、Web 前端、数据库、评估脚本五个层面。传统开发中这需要多种技能栈，但 AI 能统一处理——从 TextRank 算法实现到 Streamlit 界面布局，代码风格一致、模块接口对齐。

### 3. 调试速度

5 个关键 Bug（Python 3.13 兼容性、Ollama SDK 502、DB 路径漂移、SQLite 锁、Streamlit 缓存）都在 1-2 轮对话内定位并修复。AI 能同时分析报错堆栈、检查环境状态、提出修复方案——这些步骤传统开发中需要分别 Google 搜索、翻阅文档、论坛提问。

### 4. 文档同步生成

CONTEXT.md、ADR、README、报告、PPT 大纲——这些文档不是事后补写的，而是在开发过程中随代码同步生成。AI 利用对话历史中的决策信息，自动填充了术语表、架构描述和实验数据。

## 二、Vibe Coding 的局限

### 1. 幻觉与过度信任风险

AI 推荐的 `ollama` Python SDK 在 Windows 上存在 HTTP 502 问题，但 AI 初始代码直接使用了 SDK。如果我没有验证（运行后报错），这个 Bug 就会被遗留下来。**Vibe Coding 的前提是每次都要实际运行验证**——AI 基于训练数据的"最佳实践"不一定匹配实际环境。

### 2. 环境感知盲区

AI 无法主动感知我的系统环境。两个典型案例：
- AI 默认我将项目部署在 Python 虚拟环境中，但我实际有两个 Python（独立安装 + Anaconda），依赖安装到了错误的环境
- AI 不知道我是否安装了 Ollama——必须由我主动询问才去检查

**教训：** 环境相关的问题（工具安装、路径配置、版本兼容）需要开发者主动提出并验证。

### 3. 数据库路径这类"隐式假设"

AI 使用相对路径 `DB_PATH = "news_summary.db"` 时，默认 CWD 是模块所在目录。但 Streamlit 运行时的 CWD 是项目根目录——这导致了两处不一致。这类问题源于 AI 对不同框架（Streamlit）的运行时行为不够了解。

**教训：** 涉及跨框架集成的路径/配置问题，应主动要求 AI 使用绝对路径。

### 4. 长对话上下文衰减

在约 60 轮对话后，AI 偶尔会忘记早期的决策细节（如数据库文件的实际位置）。但通过显式检查（`查询 DB`、`cat 文件`）可以快速恢复上下文。

## 三、哪些地方需要人工介入

| 环节 | 人工介入内容 | 为什么 AI 做不了 |
|------|-------------|-----------------|
| 撰写 20 篇人工参考摘要 | 全部人工 | 需要阅读理解 + 归纳能力 + 判断什么算"好摘要" |
| 人工评分（5 篇） | 全部人工 | 需要主观判断流畅性、信息覆盖度、简洁性 |
| 分类准确率验证 | 全部人工 | 需要判断"系统分类是否合理" |
| 填写姓名/学号 | 人工 | 个人信息 |
| Ollama 安装 | 人工操作 | GUI 安装需要人工点击 |
| GitHub 仓库创建 | 人工操作 | 需要 GitHub 账户授权 |
| Git push | 人工操作 | 网络代理问题 AI 无法解决 |

**规律：** AI 能做的——代码编写、架构设计、文档生成、数据分析、调试定位。AI 做不了的——需要主观判断（摘要质量评分）、需要物理操作（安装软件、Git push）、需要个人信息（学号）。

## 四、哪些 Prompt 效果好

### 高效 Prompt 模式

**1. 选择式确认（最高效）**
> "可以" / "相信你的推荐" / "A"（从 AI 给出的选项中选）

AI 给出推荐 + 理由后，只需确认即可。这类 Prompt 占对话的 ~30%，推进速度极快。

**2. 出错后的上下文共享**
> 直接贴报错堆栈 → AI 自动定位 + 修复

不需要描述"我遇到了 XXX 问题"，只需贴完整的 traceback。AI 能从堆栈中同时看到：
- 哪个文件、哪一行
- 运行时 CWD
- Python 版本和路径
- 依赖链

这比人工描述问题高效得多。

**3. 数据驱动的下一步**
> "OK 现在导出" / "填好了" / "id=24 已填上 重新评估"

将数据交给 AI 处理，人只负责"生产数据"和"确认结果"两个环节。评估流程（ROUGE 计算、汇总统计、可视化）完全由 AI 完成。

### 低效 Prompt 模式

**"这里面有些文章太短了"** —— 没有指定哪些文章。需要 AI 追问"哪些编号？"

**教训：** 反馈越具体（带文件路径、行号、ID、具体数值），AI 修复越快。模糊反馈需要额外一轮定位对话。

## 五、总结

Vibe Coding 将开发效率提升了数倍——全栈项目从零到可运行 + 完整文档 + 实验数据，实际开发时间约 8 小时（含人工标注）。这不是"AI 取代程序员"，而是"AI 加速程序员"：代码编写、架构设计、文档生成、调试定位由 AI 完成；需求判断、质量评估、物理操作、主观决策仍由人完成。

**核心经验：**
- 每次 AI 产出的代码必须实际运行验证——不要因为"看起来对"就跳过
- 环境相关的配置（路径、依赖、工具）需要开发者主动检查
- 好的 Prompt 是"具体 + 提供上下文"（贴报错 > 描述报错）
- 人工介入集中在"AI 做不了的事"——主观判断和物理操作
