深度学习与自然语言处理 — 课程设计（毕业设计）

一、课程设计说明

1.1 课程背景

本学期课程覆盖了从基础到前沿的完整 NLP 技术栈：

模块 内容
基础层 Python、NumPy、神经网络原理、PyTorch
词表示 BPE 分词、词袋模型、Word2Vec（Skip-gram/CBOW）
序列模型 RNN、LSTM、GRU、Seq2Seq
注意力与 Transformer Attention 机制、位置编码、Multi-Head Attention、完整 Transformer
预训练模型 BERT（编码器）、GPT（解码器）、模型微调（LoRA）
AI Agent Prompt Engineering、Function Calling、ReAct、RAG、Memory、Multi-Agent、工程实践、结构化输出、评估
AI 辅助开发 Vibe Coding、Prompt 驱动开发

1.2 设计目的

课程设计是对本学期所学知识的综合运用。要求学生：

  1. 选择一个实际问题，运用课程中至少 2-3 个模块的知识解决
  2. 完成系统设计与实现，产出可运行的代码和文档
  3. 进行答辩展示，能清晰讲解技术原理和设计决策

1.3 组队规则

  • 2-3 人一组，每组选择一个题目完成
  • 报告中需明确分工，答辩时分别讲解各自负责部分
  • 鼓励互补搭配（如：一人偏模型/算法，一人偏工程/界面）

6.0 基础环境

所有题目都需要：
- Python 3.10+
- Ollama（本地运行大模型）
- 一台 8GB 内存以上的电脑（Mac/Windows/Linux 均可）

关于 GPU

大部分题目不需要 GPU。 只有涉及模型训练/微调的题目（题目 1、题目 7）需要 GPU 资源。

场景 是否需要 GPU 说明
调用 Ollama 跑 Qwen2.5 ❌ CPU 可跑，推理稍慢但完全可用
训练 LSTM/GRU ❌ 数据量小，CPU 几分钟可训练完
微调 BERT（题目 1） ⚠️ 用小模型（bert-base-chinese）CPU 可跑但慢，建议用 GPU
LoRA 微调 Qwen2.5（题目 7） ✅ 建议用 Qwen2.5-1.5B，最低需要 8GB 显存

没有 GPU 的解决方案：
1. Google Colab（免费）：每天有免费 GPU 额度，够训练小模型
2. Kaggle Notebooks（免费）：每周 30 小时 GPU，适合微调
3. AutoDL（付费）：约 1-2 元/小时，租一张 3090 跑几小时就够
4. 用更小的模型：Qwen2.5-0.5B / Qwen2.5-1.5B 在 CPU 上也能微调（慢但可行）

关于 LLM

课程设计中调用 LLM 统一使用 Ollama + Qwen2.5（本地免费运行）：

# 安装 Ollama
curl -fsSL https://ollama.com/install.sh | sh

# 拉取模型（选一个）
ollama pull qwen2.5:1.5b   # 小模型，1GB，速度快，适合开发调试
ollama pull qwen2.5:7b     # 中模型，4GB，效果好，推荐正式使用

如果效果不好，可以改为 deepseek/qwen 等其他模型调用API来完成--照道理充值几块钱就可以搞定。
微调的话，如果没有调整出效果，可以介绍做了哪些尝试，过程是什么样的。用小模型来做demo一般问题不大【Qwen2.5-1.5B】。

💡 开发阶段用 1.5b 快速迭代，最终演示用 7b 保证效果。

题目 1：电商评论情感分析看板

背景：
情感分析是 NLP 最经典的应用之一，广泛用于舆情监控、产品评价分析、客户反馈处理。本题要求学生不仅实现情感分析模型，还要构建一个完整的分析看板，让用户能导入评论数据、查看分析结果和趋势。

目标：
构建一个电商评论情感分析看板应用。用户可以导入评论数据（CSV/Excel），系统自动进行情感分析，并以可视化仪表盘展示分析结果（情感分布、时间趋势、关键词云等）。

用户使用场景：

场景 1：批量导入分析
运营小王打开系统 → 上传本月 500 条商品评论 CSV 系统显示：正在分析... (进度条) 分析完成 → 看板自动刷新： - 饼图：正面 72%、负面 18%、中性 10% - 关键词云：负面评论高频词 "物流慢"、"包装破损"、"色差" - 趋势图：本周负面评论比上周增加 5% 小王点击"负面评论"筛选 → 查看具体差评内容

场景 2：单条实时分析
小王在输入框粘贴一条新评论："包装很精美，但是用了两天就坏了，质量太差" 系统输出： - 情感判断：负面（置信度 87%） - 关键词高亮："包装很精美"(正面) / "就坏了"(负面) / "质量太差"(负面) - 分类建议：质量问题

场景 3：对比实验展示（答辩用）
切换到"模型对比"页面 → 展示： - LSTM 模型：F1=0.82，推理速度 50条/秒 - BERT 模型：F1=0.91，推理速度 10条/秒 - 同一条评论两个模型的判断差异展示

技术要求：
1. 数据准备：选择一个公开中文情感数据集，完成数据清洗、分词、划分训练/测试集
2. 方案 A（传统深度学习）：Word2Vec 词向量 + LSTM/GRU 分类器
3. 方案 B（预训练模型）：BERT 微调（冻结部分层 vs 全量微调对比）
4. 评估指标：准确率、精确率、召回率、F1-score
5. Web 界面（Streamlit/Gradio）： 支持上传 CSV → 批量分析 → 可视化展示
6. 可视化看板： 情感分布饼图、时间趋势折线图、负面评论关键词云
7. 单条分析： 输入一句话 → 输出情感判断 + 置信度 + 关键词高亮

最小可运行版本（先做到这个再加功能）：
1. 用 ChnSentiCorp 数据集训练一个 LSTM 分类器，能跑通训练+推理
2. 命令行输入一句话 → 输出正面/负面
3. 在此基础上逐步加：BERT 对比 → Web 界面 → 可视化看板

效果评估：
- 在测试集上计算 F1-score、准确率、精确率、召回率（sklearn classification_report）
- A/B 对比：LSTM vs BERT 的 F1、推理速度、参数量
- 错误案例分析：找出 5 个判断错误的样本，分析错误原因（如反讽、混合情感）
- 可选：不同训练数据量（1000/5000/10000 条）对 F1 的影响曲线

交付物：
- 训练代码 + 推理代码 + Web 应用代码
- 对比实验结果表格（至少包含：模型、参数量、训练时间、测试集 F1）
- 错误案例分析（找出 5 个判断错误的样本，分析原因）
- 可运行的 Web 应用截图/录屏

涉及课程知识： Word2Vec、RNN/LSTM、BERT、微调、PyTorch 训练流程

数据集推荐：
- ChnSentiCorp（酒店评论，正负面二分类，约 10000 条）
- 外卖评价数据集（美团/饿了么评论）
- NLPCC 2014 情感分析数据集

题目 2：基于 RAG 的个人知识库问答系统

背景：
大语言模型的知识有截止日期，且无法回答私有文档中的问题。RAG（检索增强生成）通过"先检索、再生成"的方式解决这一问题，是当前企业级 AI 应用最主流的架构。

目标：
构建一个能回答用户关于特定文档内容问题的问答系统。用户上传文档后，可以用自然语言提问，系统从文档中检索相关段落并生成回答。

用户使用场景：

场景 1：构建知识库
学生小李把本学期 NLP 课程的 10 份 PPT 和 5 篇参考论文放入 data/ 目录 运行：python main.py --build 系统输出： 📚 正在加载文档... 找到 15 个文件 📝 共切分为 342 个文本块 🔢 正在向量化... (进度条) ✅ 知识库构建完成！

场景 2：问答交互
```
小李：Transformer 中的 Multi-Head Attention 有什么作用？
系统：Multi-Head Attention 允许模型同时关注不同位置的不同表示子空间的信息。
通过将 Q/K/V 分成多个头，每个头可以学习不同的注意力模式...
[来源：第7讲_Transformer.pdf, 第12页]

小李：那和普通 Attention 有什么区别？
系统：普通 Attention 只有一组 Q/K/V，只能捕获一种注意力模式。
Multi-Head 相当于多个"视角"并行观察...
[来源：第7讲_Transformer.pdf, 第13页]
```

场景 3：检索失败的处理
小李：今天天气怎么样？ 系统：根据现有资料无法回答此问题。我的知识库中只包含 NLP 课程相关内容。 您可以尝试问课程相关的问题，比如"BERT 的预训练任务有哪些？"

技术要求：
1. 文档处理：支持导入至少 10 篇文档（PDF/Markdown/TXT），实现文本切分（chunking）
2. 向量化：使用 Sentence-Transformers 或其他 Embedding 模型将文本块向量化
3. 向量存储与检索：使用 ChromaDB 或 FAISS 存储向量，实现 Top-K 相似度检索
4. 生成回答：将检索到的上下文 + 用户问题拼接为 Prompt，调用 LLM 生成回答
5. 回答中标注引用来源（哪个文档的哪个段落）
6. 对比实验：不同 chunk 大小（256/512/1024 token）对检索质量的影响

最小可运行版本（先做到这个再加功能）：
1. 把 3 个 TXT 文件切分 → 存入 ChromaDB → 能检索到相关段落
2. 检索结果 + 问题拼成 Prompt → 调用 Ollama 生成回答
3. 在此基础上逐步加：更多文档格式 → 引用标注 → 对比实验 → Web 界面

效果评估：
- A/B 对比：有 RAG vs 无 RAG（直接问 LLM），用 LLM-as-Judge 对 20 个问题打分（1-5 分）
- 参数对比：不同 chunk_size（256/512/1024）和 top_k（1/3/5）的检索准确率
- 检索命中率：检索到的文本块是否包含正确答案（人工判断，统计命中率）
- 失败案例：找出 5 个回答不好的问题，分析原因（检索没命中？上下文不够？）

交付物：
- 完整的 RAG pipeline 代码
- 至少 10 个问答测试用例及其回答质量评估
- 对比实验报告（chunk 大小 / Top-K 数量 / 不同 Embedding 模型的效果）

涉及课程知识： 词向量/Embedding、向量数据库、RAG、Prompt Engineering

技术栈建议： ChromaDB + sentence-transformers + Ollama(Qwen2.5)

题目 3：命令行 AI 助手（ReAct Agent）

背景：
AI Agent 是 2024-2025 年最热门的技术方向。Cursor、Kiro、Devin 等产品的核心都是 Agent 架构。本题要求学生从零实现一个能理解自然语言指令并自主执行操作的命令行助手。

目标：
构建一个命令行 AI 助手，用户用自然语言描述任务，Agent 自主规划步骤、调用工具、完成任务。

用户使用场景：

场景 1：文件操作任务
```
用户：帮我把 src/ 目录下所有 .py 文件的开头加上 UTF-8 编码声明

Agent 思考：需要先列出 src/ 下的所有 .py 文件，然后逐个读取并修改
Agent 行动：[调用工具: list_directory("src/")]
观察结果：找到 main.py, utils.py, config.py
Agent 行动：[调用工具: read_file("src/main.py")]
观察结果：文件内容...（已有编码声明）
Agent 思考：main.py 已有声明，跳过。检查下一个文件
Agent 行动：[调用工具: read_file("src/utils.py")]
观察结果：文件内容...（没有编码声明）
Agent 行动：[调用工具: write_file("src/utils.py", ...)]
⚠️ 系统提示：即将修改 src/utils.py，确认？[Y/n]
用户：Y
...
Agent 回答：完成！共检查 3 个文件，其中 2 个已添加编码声明，1 个已有无需修改。
```

场景 2：信息查询任务
```
用户：这个项目用了哪些第三方库？列出来并说明各自的作用

Agent 思考：需要查看 requirements.txt 或 pyproject.toml
Agent 行动：[调用工具: read_file("requirements.txt")]
观察结果：chromadb==0.4.22, sentence-transformers==2.2.2, ollama==0.1.6...
Agent 行动：[调用工具: search_code("import chromadb")]
观察结果：在 vector_store.py 中使用
Agent 回答：
- chromadb 0.4.22：向量数据库，用于存储和检索文本向量（vector_store.py）
- sentence-transformers 2.2.2：文本向量化模型（embeddings.py）
- ollama 0.1.6：本地大模型调用（llm.py）
```

场景 3：Agent 失败的情况
```
用户：帮我部署这个项目到服务器上

Agent 思考：需要知道服务器信息，但我没有 SSH 工具
Agent 回答：抱歉，我目前没有远程连接工具，无法直接部署。
但我可以帮你：1) 生成 Dockerfile 2) 写部署脚本 3) 生成部署文档
需要我做哪个？
```

技术要求：
1. 实现 ReAct 循环：Thought → Action → Observation → ... → Answer
2. 至少实现 5 个工具：读文件、写文件、列目录、搜索代码、执行命令
3. 安全机制：写文件和执行命令前需用户确认
4. 最大步数限制（防止死循环）
5. 结构化日志：记录每一步的思考和行动，支持回放
6. 演示至少 3 个完整任务的执行过程

最小可运行版本（先做到这个再加功能）：
1. 实现 ReAct 循环 + 2 个工具（读文件、列目录）
2. 能完成一个简单任务（如"列出当前目录的文件"）
3. 在此基础上逐步加：更多工具 → 安全确认 → 结构化日志 → 复杂任务

效果评估：
- 任务完成率：准备 20 个测试任务，记录成功/失败/步数

场景 3：多 Agent 协作日志
```
[架构分析 Agent]
输入：项目文件结构
输出：{"pattern": "MVC", "entry": "app.py", "modules": ["routes/", "models/", "templates/"]}

[技术栈 Agent]
输入：requirements.txt + 代码 import 语句
输出：{"language": "Python 3.11", "framework": "Flask 2.3", "db": "SQLAlchemy+SQLite"}

[代码质量 Agent]
输入：抽样 5 个核心文件
输出：{"score": 7.2, "issues": ["routes/api.py 函数过长(120行)", "无异常处理"]}

[汇总 Agent]
输入：以上三个 Agent 的输出
输出：完整结构化报告
```

技术要求：
1. 仓库获取：通过 GitHub API 获取仓库信息、文件结构、README
2. 多 Agent 分析架构：
- 架构分析 Agent：识别项目结构、设计模式、模块划分
- 技术栈 Agent：识别语言、框架、依赖版本
- 代码质量 Agent：抽样分析代码风格、复杂度、潜在问题
- 文档 Agent：评估 README 质量、API 文档完整性
3. 报告生成：将各 Agent 分析结果汇总为结构化报告
4. Web 界面：
- 输入仓库 URL → 一键分析
- 报告可视化展示（项目结构树、技术栈标签、质量评分雷达图）
- 支持对仓库代码提问（RAG）
5. 对比功能（可选）：输入两个仓库，对比分析

最小可运行版本（先做到这个再加功能）：
1. 用 GitHub API 获取一个仓库的文件列表和 README
2. 用 LLM 分析 README + 文件结构 → 输出项目概览和技术栈
3. 在此基础上逐步加：多 Agent 分维度分析 → 代码质量评估 → RAG 问答 → Web 界面

效果评估：
- 分析准确性：5 个真实仓库，人工验证分析结果是否正确（技术栈、架构、代码规模）
- 维度覆盖率：10 个分析维度是否都有有效输出（不是空的或瞎编的）
- Agent 协作效率：各 Agent 耗时、token 消耗统计
- RAG 问答质量：对仓库代码提 10 个问题，评回答准确性
- 失败分析：哪些类型的仓库分析效果差？（如非主流语言、超大仓库）
