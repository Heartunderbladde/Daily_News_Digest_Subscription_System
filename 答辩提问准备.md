## Q1：为什么选择这个技术方案？对比过其他方案吗？

**底层回答逻辑：** 每个技术选型都有"我们选了 A 而非 B，因为 C"的清晰结构。

**逐字稿：**

> 我们的技术选型不是凭空定的，每个关键决策都有对比。
>
> **信息源**：对比过微信公众号和知乎——它们没有公开 API，需要模拟浏览器抓取，工程量大且不稳定。Hacker News 有免费 JSON API，36 氪有 RSS，都是机器可读的标准协议，零认证即可调用。两者一中一英，信息互补。
>
> **TextRank vs 其他抽取式算法**：也看过 LexRank、LSA。选择 TextRank 是因为它和课程学的 PageRank 原理直接关联，答辩能讲清楚。自己实现 80 行，每一步——TF-IDF 向量化、余弦相似度、PageRank 迭代——都能解释。
>
> **LLM 模型**：对比过 GPT-4o API 和 Qwen2.5。GPT 效果可能更好，但课程要求本地部署（Ollama），Qwen2.5:7B 在 4070 8GB 显存上刚好跑，4-bit 量化后 4.7GB，且免费。如果用 API 调 GPT，token 费累积起来不少。
>
> **Web 框架**：Streamlit vs Gradio。Gradio 适合单组件 demo（输入→输出），Streamlit 支持自由页面布局、多页导航、session state，更适合我们这种内容展示型应用。
>
> **数据库**：SQLite vs JSON 文件 vs ChromaDB。我们不需要向量检索——分类是 LLM 做的，查询主要是按日期和分类筛选——这是典型的结构化查询，SQLite 最合适。单文件、零配置、SQL 查询效率高。

### 追问：ROUGE-L 具体是怎么实现的？

如果老师追问"ROUGE 你调个库就算完了？底层怎么回事？"——用下面这个回答。

**逐字稿：**

> ROUGE-L 的核心是最长公共子序列（LCS）。给定生成摘要 X 和参考摘要 Y，找它们的 LCS 长度，然后算召回率 R = LCS长度 / |Y|，精确率 P = LCS长度 / |X|，F1 = 2RP / (R+P)。
>
> 但中文有个坑：ROUGE 标准实现（rouge-score 库）是按英文空格分词来算 n-gram 的，中文词之间没有空格，直接扔进去会把"人工智能改变世界"拆成一个个字来匹配，而不是一个个词——这没有意义。
>
> 所以我们做了一层中文分词适配：
>
> ```python
> def _tokenize_chinese(text):
>     return " ".join(jieba.cut(text))
> ```
>
> 先用 jieba 把中文切成词，用空格连接，再交给 rouge-score 库。这样 ROUGE 匹配的是"人工/智能/改变/世界"，而不是"人/工/智/能/改/变/世/界"。
>
> 整个评估流程是：
> - 读 20 篇人工参考摘要（我们自己写的）
> - 从数据库取出对应的 TextRank 和 LLM 摘要
> - 逐篇计算 ROUGE-L F1
> - 汇总求均值
>
> 说实话，ROUGE 有局限性——它只比表面 n-gram 重叠，不比语义。所以我们才补了人工评分——人工评分里 LLM 只领先 6%，但 ROUGE 领先 170%，差距这么大就是因为 ROUGE 偏向重新组织语言的长文本，而 TextRank 抽的是原文短句。这个发现本身也是实验的价值。

## Q2：模型/系统的性能瓶颈在哪？如何优化？

**底层回答逻辑：** 把瓶颈分两层——抓取层和摘要层，分别诊断和优化。

**逐字稿：**

> 性能瓶颈有两个。
>
> **第一，抓取层。** 目前是顺序抓取——50 篇文章，每篇 2-3 秒下载+提取，总共 2-3 分钟。瓶颈在 I/O 等待，不是 CPU。优化方案是用 asyncio 并发抓取——10 个并发就能压到 20 秒。Python 的 `aiohttp` + `asyncio.gather` 就能实现，改动不大。
>
> **第二，摘要层。** LLM 推理是主要瓶颈——Qwen2.5:7B 在 4070 上每篇 3-8 秒，50 篇就是 4-7 分钟。优化有几个方向：
> - 用 1.5B 小模型做快速初筛，只有重要文章才用 7B 精加工——分层摘要策略
> - 批处理：攒 5 篇文章一起发给模型——但需要调 Prompt 让模型分别输出
> - 异步队列：摘要任务扔到后台队列，不等它完成就返回页面——用户看到的摘要先由 TextRank 兜底，LLM 版本生成了再替换
>
> **TextRank 本身不是瓶颈**——纯 CPU 计算，一篇不到 1 秒。

## Q3：如果数据量增大 10 倍，架构需要怎么调整？

**底层回答逻辑：** 从"单机玩具"到"可扩展系统"的分层改造。

**逐字稿：**

> 10 倍意味着从每天 30-50 篇涨到 300-500 篇。架构需要三层改造：
>
> **抓取层**：现在顺序抓取，10 倍后等不了。改为生产者-消费者模式——一个 fetcher 线程获取文章 URL 放进队列，多个 worker 线程并发下载正文。Python 的 `concurrent.futures.ThreadPoolExecutor` 就能搭。
>
> **存储层**：SQLite 在单机万级数据没问题，但并发写入是瓶颈。如果量再大到十万级，换 PostgreSQL。表结构不变——我们文章和摘要的设计已经是规范化的。
>
> **摘要层**：这是最大挑战。500 篇文章全部走 LLM 推理不现实——7B 模型每篇 5 秒就是 40 分钟。方案是**分层策略**：
> - 第一层：TextRank 快速过全部 500 篇（不到 1 分钟）
> - 第二层：用 1.5B 小模型过滤，只保留"高价值"文章（基于热度分数、关键词匹配）
> - 第三层：7B 大模型精加工 Top 30 篇
>
> **Web 展示层**：加分页而不是全量渲染——Streamlit 原生不支持分页，但可以用 `st.selectbox` 切页。
>
> 另外，加一个监控——Prometheus 或者简单的 Python logging——跟踪每天抓取成功率、摘要生成率、LLM 调用失败率。

## Q4：这个项目最难的部分是什么？

**底层回答逻辑：** 不是诉苦，而是展示解决问题的能力。

**逐字稿：**

> 最难的不是写 TextRank 算法，也不是调 Streamlit 界面——这两个都是确定性的，写了就能跑。
>
> **最难的是 Prompt 工程。** 让 LLM 稳定输出 JSON 格式、同时完成分类和三种长度摘要，远比想象中复杂。
>
> 刚开始 LLM 的输出很不稳定——有时不返回 JSON 而是文章评价，有时输出的分类标签不在我们的四分类里，有时 JSON 里漏了字段。调了好几版 Prompt：
> - 第一版只说了"请生成摘要"，输出格式完全随机
> - 第二版加了"输出 JSON"，但字段名有时候是 `one_sentence` 有时候是 `sentence_one`
> - 最终版才稳定——明确指定 JSON 结构模板、列出四个分类标签、temperature 降到 0.3、输出截断保护
>
> 另一个麻烦是**环境兼容性**。Python 3.13 移除了 `sgmllib` 标准库，导致 newspaper3k 和 feedparser 两个库直接不可用。排查了两三个小时，最后决定整条抓取链换——正文提取从 newspaper3k 换到 trafilatura，RSS 解析从 feedparser 换到 lxml 直接解析。这种问题不是算法问题，但一样耗时。
>
> 最有意思的是 Ollama Python SDK 的 502 bug——`curl` 直接调 API 正常，SDK 却报错。最后直接用 `requests` 库替代了官方 SDK，一行 `requests.post` 就解决了。

## Q5：课程中哪些知识点在项目中用到了？

**底层回答逻辑：** 用表格快速过一遍，然后挑 2 个深入讲。

**逐字稿：**

> 我把涉及的知识点分成三层来梳理——基础层、模型层、工程层，各对应课程不同模块。
>
> **基础层**（词表示模块）：
> - jieba 分词 + 停用词过滤：TextRank 的第一步
> - TF-IDF 词向量：TextRank 的句子向量化核心，不是简单词频
> - 余弦相似度：衡量句子之间的语义距离
>
> **模型层**（序列模型 + Transformer 模块）：
> - PageRank 图算法：TextRank 的理论基础（和课程学的注意力权重有相似之处——都是在图上做信息传播）
> - Transformer 架构：这是 Qwen2.5 大模型的底层原理，Multi-Head Attention 和位置编码课上讲过
> - Seq2Seq 概念：LLM 生成式摘要本质上是"输入长文章 → 输出短摘要"的序列到序列任务
>
> **工程层**（AI Agent + 工程实践模块）：
> - Prompt Engineering：一次性完成分类+多长度摘要，结构化 JSON 输出
> - Function Calling 思想：虽然没直接用 Function Calling，但让 LLM 输出 JSON 本质上就是让模型按规定的接口输出
> - 模块化架构：6 个模块解耦，fetcher 和 summarizer 可以独立替换
> - SQLite 持久化 + Streamlit 前端
>
> 总共覆盖了课程 7 个模块中的至少 5 个——词表示、序列模型、Transformer、Agent、工程实践。

## 补充问题（老师可能追问）

### Q6：TextRank 和 LLM 摘要如果结论矛盾怎么办？

> 这是一个好问题，我们没有遇到过 TextRank 和 LLM 结论矛盾的场景——两者的差异不在于"对错"，而在于形式。TextRank 是抽取原文句，LLM 是重新归纳。
>
> 如果真遇到矛盾（比如 TextRank 抽了支持观点 A 的句子，LLM 归纳出观点 B），我们的策略是以 LLM 为准——因为它理解语义而非机械抽取。但我们会把两套摘要都展示出来（实验对比页面就是这个目的），让用户自己判断。

### Q7：你自实现的 TextRank 到底对不对？怎么验证？

> 我们做过正确性交叉验证——拿同一篇文章，用我们的 TextRank 和 sumy 库（虽然生产没用但 debug 时装过）对比输出。结果：Top-3 句中至少有 2 句重叠，平均重叠率超过 70%。差异主要在停用词过滤策略和分句粒度上，不是算法错误。
>
> 更好的验证方式其实是人工评分——我们的 TextRank 流畅性 5.0、信息覆盖度 4.0——如果算法完全不对，这两个分不会这么高。

### Q8：LLM 生成的摘要会不会有事实错误（幻觉）？

> 这是我们设计时考虑过的问题。为了减少幻觉我们做了几件事：
> - temperature 设为 0.3，降低随机性——模型不会"自由发挥"
> - System Prompt 明确要求"摘要必须基于原文事实，不要编造信息"
> - 正文截断到前 3000 字符——不给模型"发挥空间"
>
> 从我们的人工评分来看，LLM 在 5 篇测试中信息覆盖度 4.8/5，没有发现明显的事实错误。但如果真正上线，仍然应该保留"人工抽查+用户反馈"的机制。

### Q9：为什么用 7B 而不是 1.5B 或更大的模型？

> 1.5B 在开发早期试过——速度很快，但摘要质量不稳定，对英文文章的翻译摘要能力弱。7B 在 4070 8GB 显存上刚好跑 4-bit 量化版，速度能接受（3-8 秒一篇），质量明显更好。
>
> 更大模型（14B、32B、72B）要么显存放不下，要么推理太慢不适合实时场景。7B 是这个硬件的甜点。

### Q10：如果让我重新做这个项目，会改什么？

> 我会做两个改进：
>
> 第一，加异常监控。现在 LLM 调用失败是静默跳过的，应该加一个简单的指标统计——每天调了多少次 LLM、成功率、平均耗时——方便排查问题。
>
> 第二，把"刷新数据"和"生成摘要"解耦成异步操作。现在用户点刷新要等几分钟，应该设计成后台任务——用户先看到 TextRank 版本，LLM 版本慢慢更新。类似搜索引擎"爬取-索引"的分离设计。
